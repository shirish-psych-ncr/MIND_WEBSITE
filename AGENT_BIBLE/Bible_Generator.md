# Engineering Knowledge Base as an AI Cognitive Architecture

## Fundamental Premise

The repository is not the project;;the repository is the externalized memory of the project;;source code is merely one representation of system knowledge;;documentation is not commentary on code but the canonical representation of engineering intent;;every file exists to reduce future uncertainty;;every document exists to eliminate a future question;;every engineering artifact should answer "why", "what", "where", "when", "who owns it", "what depends on it", "what breaks if it changes", and "how should future agents modify it".

---

## Cognitive Layer Separation

Never organize documentation by technologies;;organize by cognitive permanence;;separate timeless knowledge from slowly evolving knowledge from rapidly changing knowledge;;architecture should survive framework rewrites;;design philosophy should survive redesigns;;component specifications should survive implementation changes;;inventories should never contain reasoning;;reasoning should never contain inventories;;project memory should never become architecture;;temporary planning should never pollute permanent specifications.

---

## Knowledge Gravity

Every engineering concept naturally pulls related information toward itself;;allow every concept exactly one gravitational center;;all other documents orbit it through references;;never allow competing centers of truth;;if two documents explain the same thing one is already obsolete;;AI should spend more effort discovering existing gravity than creating new documents.

---

## Information Compression

Documentation should compress understanding rather than compress words;;a paragraph should replace hundreds of implementation details;;a diagram should replace pages of prose;;a registry should replace directory traversal;;a specification should replace source-code archaeology;;avoid documenting symptoms when underlying principles explain them;;teach generators not instances;;describe systems not examples;;record invariants instead of implementations.

---

## Information Expansion

Every abstract rule should be expandable into implementation;;every implementation should collapse back into an abstract rule;;architecture expands into systems;;systems expand into modules;;modules expand into components;;components expand into pages;;pages expand into interactions;;interactions expand into events;;events expand into state transitions;;state transitions expand into implementation;;reverse traversal should always reconstruct higher-level intent.

---

## AI Context Loading

AI should never begin by opening code;;first reconstruct the project mentally;;understand why the repository exists;;infer product philosophy;;identify architectural boundaries;;identify stability boundaries;;identify ownership boundaries;;predict folder purposes before inspecting them;;predict component behaviour before reading implementation;;only then verify assumptions against source code;;implementation should validate understanding rather than create it.

---

## AI Documentation Loop

Every implementation task begins as hypothesis formation;;identify affected knowledge domains;;predict required documentation changes before modifying code;;modify implementation;;validate architectural consistency;;update permanent knowledge;;update transient memory;;regenerate inventories;;remove redundancy introduced during development;;leave repository easier to understand than before the change.

---

## Stable versus Volatile Knowledge

Stable knowledge deserves prose;;volatile knowledge deserves automation;;inventories should always be generated;;statistics should always be generated;;dependency maps should always be generated;;asset counts should always be generated;;performance metrics should always be generated;;only decisions deserve manual writing;;only rationale deserves human language;;numbers should never be maintained manually if derivable.

---

## Documentation as Dependency Graph

Every document should know why it exists;;what produces it;;what consumes it;;what validates it;;what supersedes it;;what depends upon it;;which other documents become invalid if it changes;;AI should reason about documents exactly as dependency injection containers reason about software modules.

---

## Evolution Rather Than Mutation

Never rewrite documentation without preserving evolutionary history;;record architectural migrations;;record discarded approaches;;record assumptions proven false;;record rationale behind reversals;;future AI should understand why present architecture exists instead of rediscovering abandoned ideas.

---

## Repository Self-Awareness

Repository should continuously describe itself;;enumerate its own pages;;enumerate its own components;;enumerate its own dependencies;;enumerate its own architectural layers;;enumerate performance characteristics;;enumerate accessibility coverage;;enumerate testing maturity;;enumerate technical debt;;enumerate unfinished migrations;;AI should never manually answer questions the repository can answer automatically.

---

## Documentation Stratification

Strategic documents answer "why does this system exist?";;architectural documents answer "how is this system organized?";;engineering documents answer "how is this implemented?";;operational documents answer "how is this maintained?";;generated documents answer "what currently exists?";;historical documents answer "how did this evolve?";;memory documents answer "what changed recently?";;temporary planning documents answer "what happens next?";;mixing strata destroys discoverability.

---

## AI Ownership Model

AI should behave as curator before engineer;;librarian before programmer;;historian before optimizer;;architect before implementer;;maintainer before feature developer;;documentation debt is architectural debt;;knowledge fragmentation is technical debt;;duplicate explanations are merge conflicts waiting to happen.

---

## Living Registries

Registries should never explain;;they enumerate;;Specifications should never enumerate;;they define;;Guides should never prescribe architecture;;they teach usage;;Protocols should never explain implementation;;they coordinate behaviour;;Memory should never become documentation;;documentation should never become memory;;every document should refuse responsibilities outside its domain.

---

## Cross-Link Density

Documents should reference concepts rather than repeat them;;prefer semantic relationships over textual duplication;;every concept should expose upstream dependencies;;downstream consumers;;adjacent systems;;validation mechanisms;;migration paths;;AI should navigate concept graphs rather than filename trees.

---

## Change Propagation

Before changing code AI predicts documentation impact;;before changing documentation AI predicts implementation impact;;before deleting files AI predicts dependency impact;;before adding files AI predicts ownership impact;;every modification begins by identifying ripple effects;;repository consistency is maintained by reasoning about propagation rather than isolated edits.

---

## Architectural Memory

Project memory is not chat history;;it is accumulated engineering state;;capture unresolved tensions;;known trade-offs;;temporary compromises;;pending refactors;;assumptions awaiting validation;;active migrations;;experimental branches;;future AI sessions should inherit engineering momentum rather than conversational memory.

---

## Documentation Entropy

Entropy increases through duplication;;inconsistent terminology;;manual inventories;;implicit assumptions;;undocumented shortcuts;;orphaned specifications;;obsolete examples;;abandoned migrations;;AI maintenance should primarily fight entropy rather than produce volume;;deleting obsolete knowledge is as valuable as creating new knowledge.

---

## AI Success Criteria

Successful AI assistance is measured by reduced ambiguity;;reduced onboarding time;;reduced architectural drift;;reduced duplicated knowledge;;increased predictability;;increased discoverability;;increased automation;;increased semantic consistency;;increased repository self-description;;increased confidence that an unfamiliar engineer or autonomous coding agent could reconstruct project intent solely from repository knowledge without external explanation.

---

## Ultimate Objective

The ideal repository behaves like an executable knowledge system;;source code implements behaviour;;documentation implements understanding;;automation synchronizes reality;;AI maintains coherence;;Git preserves evolution;;future contributors inherit reasoning instead of merely inheriting files;;every engineering decision becomes discoverable;;every architectural assumption becomes explicit;;every implementation becomes traceable back to design intent;;every future AI session starts with comprehension instead of reconstruction.
